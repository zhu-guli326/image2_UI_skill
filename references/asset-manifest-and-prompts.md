# 资产清单与提示词指南

规划 UI 实现中的 Image2 图片资产时，使用这份参考。

## 前期审查文档模板

用户提供 UI 图后，先输出这类文档，再进入生图。

```markdown
## UI 还原前期审查

### 1. 整体判断
- 页面类型：
- 主要视觉风格：
- 核心布局：
- 首屏重点：
- 主要风险：

### 2. 元素还原难度

| 区域/元素 | 建议实现方式 | 难度 | 原因 | 是否需要用户确认 |
| --- | --- | --- | --- | --- |
| 顶部导航 | 代码 | 容易 | 文本和按钮结构清晰 | 否 |
| 首屏背景图 | Image2 | 困难 | 依赖复杂摄影/插画氛围 | 是 |
| 品牌 logo | 使用原素材 | 不建议直接生成 | 需要精确品牌一致性 | 是 |
| 标题字体 | Web 字体/授权字体 | 中等 | 字体气质影响还原度 | 是 |

### 3. 字体判断

| 用途 | 参考图气质 | 建议字体方案 | 授权/来源 | fallback | 风险 |
| --- | --- | --- | --- | --- | --- |
| 标题 | 几何无衬线、较粗、字面宽 | 项目已有字体或开源近似字体 | 待确认 | system-ui, sans-serif | 字重和中文字形可能不完全一致 |
| 正文 | 清晰中性无衬线 | 系统字体栈 | 系统内置 | system-ui, sans-serif | 跨平台显示略有差异 |

### 4. 图片生成候选

| id | UI 位置 | 用途 | 预估槽位 | 建议导出尺寸 | 是否透明 | 风险 |
| --- | --- | --- | --- | --- | --- | --- |
| hero-bg | 首屏 | 背景氛围图 | 100vw x 60vh | 2880x1600 | 否 | 构图需给标题留白 |

### 5. 需要确认
- 是否需要 100% 还原原图，还是允许风格近似？
- 是否有 logo、产品图、人物照片等原始素材？
- 是否有品牌字体、商业字体或指定字体文件？是否允许使用开源近似字体？
- 是否需要同时适配移动端？
- 是否允许使用外部抠图 API？如果不允许，优先使用 Image2 自带能力或本地工具。
- 最终页面要实现在哪个技术栈/项目目录？

### 6. 下一步建议
1. 先确认品牌素材和页面尺寸。
2. 再生成首屏/背景/产品图等核心图片资产。
3. 集成后做桌面端和移动端页面级审查。
```

难度判断：

- `容易`：代码或现有图标库即可稳定还原。
- `中等`：需要精细 CSS、响应式处理、图标替换或轻量图片辅助。
- `困难`：需要 Image2、生图迭代、抠图、复杂质感、摄影/插画匹配或精细裁剪。
- `不建议直接生成`：精确 logo、商标、用户专属照片、版权素材、精确产品截图。

## 风险排查清单

前期审查时补充这些判断：

| 风险 | 典型表现 | 处理方式 |
| --- | --- | --- |
| 目标尺寸不明 | 只有参考图，没有最终页面宽度/设备 | 询问目标端，或按桌面 1440px 与移动 390px 做假设 |
| 还原精度不明 | 用户没说明像素级还是风格参考 | 询问“精确复刻还是风格近似” |
| 缺少原始品牌素材 | logo、产品图只出现在截图里 | 要求原素材；不要直接生成精确 logo |
| 字体授权不明 | 参考图像品牌字体或商业字体 | 要求用户提供授权字体文件，或使用开源/系统近似字体 |
| 字体加载异常 | 字体闪烁、缺字、按钮变宽、行高跳动 | 设置 fallback、`font-display`、子集化，并做渲染检查 |
| 模型生成伪文字 | 图片里出现乱码、字母、水印 | 提示词禁止文字，验收时检查，必要时重生成 |
| 多资产风格漂移 | 首屏图、卡片图、插画不像同一套 | 统一风格 token：色彩、光照、材质、视角、线条 |
| 抠图质量差 | 白边、硬边、残留背景、阴影脏 | 换背景重抠，或用本地/授权 API，验证 alpha |
| 移动端裁剪失败 | 主体被裁掉，文案没有留白 | 单独生成移动端裁剪，或调整 `object-position` |
| 性能过重 | 首屏图片过大，PNG 太多 | 压缩 WebP/AVIF，使用 `srcset`、懒加载和明确尺寸 |
| 可访问性差 | 文字在图片里，alt 缺失 | UI 文本用代码渲染，补充合适 alt |
| 线上资源丢失 | 本地能显示，部署后图片 404 | 检查资源目录、大小写、构建产物和网络请求 |
| 外部 API 风险 | 需要上传用户图片去抠图 | 先获得用户许可并确认凭据；敏感资产使用本地方案 |

如果风险无法通过合理假设解决，先向用户确认。优先确认会影响版权、隐私、品牌准确性、最终尺寸和技术栈的问题。

## 资产清单

生成前先创建资产清单。多数情况下，一个紧凑表格就够：

| id | UI 位置 | 类型 | 代码或 Image2 | CSS 槽位尺寸 | 导出尺寸 | 比例 | 提示词摘要 | 后处理 | 目标路径 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| hero-bg | 首屏背景 | 首屏图 | Image2 | 100vw x 56vw | 2880x1600 | 1.8:1 | 编辑风产品场景 | 裁剪、压缩 | src/assets/generated/hero-bg-2880.webp |

可使用的资产类型：

- `hero-image`：首屏图
- `background-plate`：背景底图
- `product-render`：产品渲染图
- `photo`：照片
- `illustration`：插画
- `avatar`：头像
- `thumbnail`：缩略图
- `texture`：纹理
- `cutout`：抠图
- `custom-icon`：自定义图标

可使用的后处理类型：

- `none`：无需处理
- `crop`：裁剪
- `resize`：缩放
- `remove-background`：去背景
- `mask`：遮罩
- `transparent-png`：透明 PNG
- `compress-webp`：压缩 WebP
- `mobile-crop`：移动端裁剪

## 尺寸规划

对每个资产：

1. 测量或推断它在参考截图中的槽位。
2. 转换为实现中的 CSS 尺寸。
3. 选择导出尺寸：
   - 很小或低优先级装饰图可用 1x。
   - 普通 UI 图片通常用 2x。
   - 清晰前景抠图可用 2x 到 3x。
4. 匹配参考图时保留原始宽高比。
5. 如果桌面端和移动端需要不同构图，生成单独裁剪。

常用起始尺寸：

| 用途 | 建议导出尺寸 |
| --- | --- |
| 桌面端首屏背景 | 2880x1440、2880x1600 或 3200x1800 |
| 移动端首屏背景 | 1440x1920 或 1536x2048 |
| 大型产品抠图 | 1600x1200 或 1800x1800 PNG/WebP alpha |
| 卡片缩略图 | 800x600 或 1024x768 |
| 头像/肖像 | 512x512 或 1024x1024 |
| 空状态插画 | 1200x900 |
| 纹理/背景平铺 | 1024x1024 或 2048x2048 |
| 自定义图标 | 512x512，必要时再转换/简化成矢量 |

```text
为一个 [产品/网站/App] UI 生成 [资产类型]。

用途和位置：
- 用于 [UI 槽位]。
- 目标宽高比：[比例]。
- 目标导出尺寸：[宽]x[高]。

主体和构图：
- [主要主体]
- [视角/镜头/画面框取]
- [前景与背景关系]
- [文案留白需求]

风格：
- [真实摄影/编辑风/3D/扁平插画等]
- 色彩：[颜色]

- 质感/材质：[细节]

集成约束：
```

## 提示词示例

首屏背景：

```text
为现代效率工具 SaaS 首页生成一张宽幅编辑风首屏背景图。
目标宽高比 16:9，导出尺寸 2880x1620。
画面展示优雅桌面场景：一台笔记本电脑、柔和晨光、少量纸张和一盆小绿植。
在左上三分之一区域保留干净留白，方便叠加标题文字。
风格为高级真实摄影，中性白、石墨灰、低饱和绿色点缀，浅景深。
不要出现可读文字、logo、水印、UI 标签、手或人物。
```

产品抠图：

```text
生成一个真实感悬浮产品抠图，主体是一台小型智能音箱。
目标尺寸 1600x1600 PNG，透明背景。
三分之四正面视角，居中构图，柔和棚拍光线，可见织物纹理，保留自然柔和阴影为 alpha。
不要出现文字、logo、标签、水印、背景场景或额外物体。
```

卡片缩略插画：

```text
为仪表盘分析卡片生成一张紧凑插画。
目标宽高比 4:3，导出尺寸 1024x768。
抽象数据可视化，由层叠玻璃面板和小型彩色图表形状组成。
风格干净、半 3D、边缘清晰，白色背景，点缀色为青绿色、珊瑚色和石墨灰。
不要出现可读数字、文字、logo、UI 控件或水印。
```

## 抠图检查清单

去背景前：

- 优先让主体不要贴到画布边缘。
- 如果生成时不能透明背景，要求简单且对比明确的背景。
- 阴影保持独立柔和；避免在头发、玻璃、细毛、烟雾或透明物体后面放复杂背景。

去背景后：

- 验证文件确实有 alpha 通道。
- 在浅色和深色背景上检查边缘光晕。
- 如果 UI 需要自然阴影，保留柔和阴影。
- 需要无损 alpha 时导出 PNG；文件体积更重要且质量可接受时导出 WebP。

## 页面级审查清单

生成并集成图片后，检查完整页面：

- 用实际浏览器查看，不只看静态代码。
- 至少检查桌面端和移动端两个视口。
- 检查页面真实文字是否乱码、截断、溢出或被图片遮挡。
- 检查生成图片内部是否出现伪文字、乱码、logo、水印或错误 UI 元素。
- 检查图片是否拉伸、压扁、模糊、裁掉主体或比例错误。
- 检查图片和代码 UI 的融合：色彩、光照、阴影、边缘、透视和清晰度是否统一。
- 检查抠图边缘：是否有白边、硬边、脏边、透明区域残留或不自然阴影。
- 检查层级和交互：图片是否遮挡按钮、链接、表单控件或 hover/click 区域。
- 检查移动端构图：主体是否仍可见，文案留白是否仍存在，是否需要移动端专用裁剪。
- 如果页面明显不自然，优先修正最影响整体观感的问题，再重复截图审查。

## 原图差距核对表

最终页面截图后，用这张表对照原始参考图。每轮修改后都更新一次。

| 轮次 | 区域 | 原图表现 | 当前实现 | 差距等级 | 修正动作 | 结果 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 首屏图片 | 主体在右侧，左侧留白充足 | 主体偏中，标题空间被压缩 | 必须修 | 调整裁剪或重生成图片 | 待复查 |
| 1 | 标题字体 | 较宽的几何无衬线，字重 700 | 当前字体偏窄 | 建议修 | 替换近似 Web 字体或调字重 | 待复查 |
| 1 | 卡片间距 | 卡片间距紧凑 | 间距偏大 | 可接受差异 | 暂不处理 | 接受 |

差距等级：

- `必须修`：明显影响还原度、可用性或用户明确要求的核心视觉。
- `建议修`：影响质感，但不阻碍整体交付。
- `可接受差异`：受素材、授权、模型质量、响应式适配或用户选择限制，可以记录保留。

每轮核对时至少检查：

- 布局比例：首屏高度、左右留白、区块间距、网格比例。
- 字体：字号、字重、行高、字间距、中文/英文/数字显示。
- 色彩：背景、按钮、文字、边框、阴影、透明度。
- 图片：主体位置、裁剪、风格、光照、边缘、清晰度。
- 组件：按钮、卡片、图标、输入框、导航状态。
- 响应式：桌面端和移动端是否都接近参考图意图。

## 集成检查清单

- 使用明确尺寸或宽高比，避免图片加载时页面跳动。
- 确认图片在最大渲染尺寸下不模糊。
- 确认生成图片里没有与真实 UI 文案冲突的文字。
- 确认前景抠图不会意外遮挡可点击控件。
- 确认移动端裁剪仍保留主体和文案留白。
- 确认字体文件能正常加载，fallback 不会导致布局明显变化。
- 确认中文、英文、数字、按钮文字都没有乱码、缺字、截断或溢出。
- 确认字体授权来源已记录，未授权商业字体没有被嵌入项目。
